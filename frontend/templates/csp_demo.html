Background - what CSP does and how it differs from CORS

CSP is an instruction the server gives the browser: "for this document, only execute scripts / load images / connect to origins that match this allowlist". It is enforced entirely on the client - if the browser cooperates, even an attacker who succeeded in injecting markup into the page is stopped from running their payload.

CSP vs CORS - they are independent.

  • CORS governs whether JavaScript on the page is allowed to read a cross-origin response. The request still leaves the browser; the server decides via response headers.
  • CSP governs whether the browser is allowed to send the request / execute the resource at all, based on the policy of the page that issued it. CSP runs before CORS - if CSP blocks, CORS never even comes into the picture.

In the four modes below, the API and evil servers are unchanged - the only thing that changes is the policy this page itself declares.

Test 1 - script-src

Load <script src="http://evil.local:8082/evil.js">

Simulates an attacker-supplied script being injected into the page (e.g. via a vulnerable third-party widget). With no CSP, the browser happily fetches and executes it. With default-src 'self' or script-src 'nonce-...', the browser refuses to even transmit the request.

      Demonstrates connect-src, which controls

    Test 4 - inline handlers / DOM XSS
    
innerHTML = '<img src=x onerror="...">'

    

      Simulates a stored / reflected XSS payload reaching the DOM. CSP
      without 'unsafe-inline' in script-src blocks
      every inline event handler (onerror, onload,
      onclick, ...) regardless of how the markup got there.
      This is the single biggest reason CSP exists.
