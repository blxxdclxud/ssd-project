CORS Attack Demo

This page simulates an attacker's site attempting to steal data from api.local:8081 using cross-origin requests from a different origin.

What this demonstrates

When a server is correctly configured (CORS disabled or whitelisting only frontend.local:8080), requests from evil.local:8082 are blocked by the browser - the attacker cannot read any data even though the HTTP request may have reached the server.

When a server is misconfigured (CORS_MISCONFIG=1), it blindly reflects the requesting Origin header back as Access-Control-Allow-Origin and sets Access-Control-Allow-Credentials: true. The browser then permits evil.local to read the full response, including sensitive endpoints.

Note: using Access-Control-Allow-Origin: * combined with credentials would be rejected by browsers outright. Reflecting the specific origin is the real-world pattern that actually works as an attack vector.

Attack 1 - Public data

Steal GET /api/data

Fetches http://api.local:8081/api/data from evil.local:8082. With correct CORS the browser blocks the read. With CORS_MISCONFIG=1 the full user list is returned to this page. 


  


  
  

    Attack 2 - Private credentials
    
Steal GET /api/private

    

      Fetches the private endpoint http://api.local:8081/api/private.
      This endpoint returns an admin token. With CORS_MISCONFIG=1 evil.local
      can read it in full.
    

    
    

  


  
  

    Attack 3 - Credentialed request
    
Steal GET /api/private with credentials: include

    

      Same as Attack 2 but with credentials: 'include', which instructs
      the browser to send any cookies or auth headers along with the request.
      This requires both Access-Control-Allow-Origin to be the exact
      requesting origin (not *) and
      Access-Control-Allow-Credentials: true in the response.
      The CORS misconfig satisfies both conditions.
